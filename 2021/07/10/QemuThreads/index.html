Qemu Threads

1. 事件循环和qemu的线程

1.1 事件循环（Event loop）

事件循环是一种设计模式，它等待和分配事件或消息（waits and dispatches）。他的工作原理是：主循环初始化资源，然后阻塞请求直到有事件到达（waits）。事件到达后对事件进行监听（poll），当事件的预处理完成后（例如等待一段时间、获取一个packet、写入一个文件），poll返回，返回后调用该事件对应的callback function（dispatch）。

下图是针对事件循环机制所设计的演示图。在事件循环的运行中，第一步是完成资源的初始化（prepare），填充一个事件文件描述符的列表，然后监听对应的文件描述符。等到文件描述符的状态变为 ready 时，将其放入事件队列（poll）。主循环中的代码执行完成后（即栈清空），将事件队列中的事件逐次地放入程序调用栈中调用其callback function（dispatch）。

1.2 QEMU的线程

QEMU有几种不同类型的线程：

  1. vCPU线程，模拟CPU中的一个核
  2. 主循环（qemu/softmmu/main-loop.c）
  3. IO线程，在运行vCPU和外带QMP监控命令的同时运行模拟设备的事件循环
  4. Worker thread（VNC、SPICE……）

VNC是worker thread的一个例子，他用了一个专门的worker thread（ui/vnc-jobs.c）来进行计算密集型的图像压缩和编码工作。

1.3 主循环线程

当guest code（即虚拟机上的程序）访问设备寄存器的时候，vCPU线程捕获这个事件，将其分配给一个模拟设备，设备的read/write方法运行在vCPU线程上。 此时vCPU线程被阻塞，直到设备工作结束并返回。这样的方法意味着CPU将会被IO阻塞，尤其是那些需要较长时间运行的设备，例如磁盘。

许多设备为了解决这个问题，采用了主循环线程的事件循环来代替vCPU循环。这些设备在主循环中加入计时器或是文件描述符（file descriptor）来监控回调函数，然后返回主程序执行循环下一条命令（guest code）。当计时器时间到了（timer expires）或文件描述符变为可用状态（file descriptor becomes ready）时，回调函数在主循环中运行。

主循环线程负责分配（dispatch）两种类型的任务，每种任务都有对应的多种事件源:

  1. File descriptor event
    • aio: block I/O, ioeventfd
    • iohandler: old event handler inside the qemu (net, nbd, audio, ui, vfio, …)
    • slirp: -net usr
    • chardev: -chardev xxx
  2. Non-fd services
    • timer
    • bottom halves

主循环分为三个部分，preparation，poll和dispatch。准备阶段处理文件描述符的注册，用一个数组将文件描述符保存起来。poll阶段接收各种事件，将他们放入队列中，在dispatch阶段将其分配给回调函数执行。

1.4 vCPU线程与IO事件处理

1.4.1 kvm基本原理

系统虚拟机提供完整的设备模拟、运行系统和相互隔离，为了管理好这些虚拟机，我们需要在Host中提供一个统一的管理机制，即管理全局物理资源的资源管理器——VMM（Virtual Machine Monitor）。不管是qemu还是kvm，都是一种VMM，但是他们的实现方式完全不同。

以qemu为基础的虚拟机是对所有硬件进行软件层面的完全模拟、并利用tcg将guest code转化成中间代码，再根据中间代码进行处理、执行。而这样的方法会导致效率很低。

而KVM的实现是基于硬件实现的。KVM利用Intel和AMD支持的x86平台下的硬件虚拟化机制实现，并成功地进入到Linux内核中，成为Linux的内核模块。硬件层面的虚拟化机制实际上是提供了全新的客户机操作模式（A new guest operating mode）、硬件层面的状态转换（Hardware state switch）及退出原因报告机制（Exit reason reporting）。这三个重要的硬件实现为kvm虚拟机的运行提供了基础。

kvm启动虚拟机后，虚拟机成为host中的一个普通的进程。进程对guest code进行执行，此时硬件转到guest mode。这个模式下能够选择性地使某些操作（例如对特定寄存器的访问）陷入内核中。下图[12]是kvm执行时在三个模式中切换的示意图。

kvm能够让guest code的执行充分利用host的CPU和内存，但对于更多的设备，尤其是IO设备，就需要借助qemu的既有实现。qemu在上图中就相当于是执行在user mode的软件，kvm则执行在kernel mode，负责对事件进行调度，qemu的vCPU线程实际上就是通过ioctl的系统调用进入到guest mode执行guest OS的。

qemu-kvm设备模拟有三种方式，分别是纯软件的设备模拟、virtio设备模拟和直通设备(VFIO)。

virtio的用途主要是磁盘设备和网络设备这两类。根据以往的研究成果可以发现，在虚拟机的效率方面，磁盘和网络的读写影响巨大，而之前单纯的设备模拟机制只能提供相对来说不够强大的性能，为了提高这部分设备的性能，virtio应运而生。但为了实现virtio，我们必须在guest上安装对应的驱动，这产生了额外的开销，因此我们称这种虚拟化方式为半虚拟化（para-virtualization）。

1.4.2 virtio事件的处理流程

vCPU线程最主要的功能将guest code转化成host code执行。当IO事件发生的时候，vCPU线程会引发VM exit，退出到KVM中，KVM负责判断自身能否处理，如果不能，则将事件交由qemu进行处理。这个过程中，如果访问的地址空间被标特殊的ioeventfd所绑定，那么KVM会利用eventfd()系统调用对该eventfd发出signal，对应的eventfd的状态发生改变，之后直接返回guest mode继续执行，即vCPU直接返回。

这里的ioeventfd与一个地址addr绑定，地址addr在qemu中被封装成MemoryRegion。在绑定的过程中，可以选择是完全匹配还是任意匹配。所谓完全匹配就是虚拟机向地址写入的值与预先设定的值相同时才触发ioeventfd，否则使用通用的IO分发处理。而任意匹配则不考虑虚拟机写入的值。

而qemu事件循环正在poll过程中等待，发现eventfd状态改变之后poll返回，进行check、dispatch等处理。在qemu事件循环的处理过程中，事件循环会使用系统调用io_submit()来调用对应实际设备的驱动，由驱动再控制实际的设备，实际设备开始异步工作，系统从内核态返回用户态，由qemu继续执行下一步的操作（例如进行下一次的事件循环）。这样就实现了一次IO事件到qemu事件循环的分配，这中间不需要中途切换到用户态qemu的vCPU线程执行，只需要qemu事件循环对eventfd响应之后进行IO处理即可。

事件完成后，事件返回会遵循一个相反的方向。设备驱动发送事件完成的消息，KVM收到后向特定的eventfd发送signal，qemu事件循环进行后续处理，结束后返回KVM。KVM朝vCPU线程发送中断，表示此时IO已经完成。

1.4.3 ioeventfd的注册

qemu中eventfd会与特定的MemoryRegion进行绑定，这里我们需要研究清楚qemu中ioeventfd的注册机制，从而便于我们在实际程序中验证。

首先，ioeventfd的完整注册机制分成两部分：qemu的注册和kvm的注册。qemu的注册主要是为了将特定的eventfd与对应的I/O地址绑定起来；kvm的注册则是为了在kvm处理IO时能够将特殊的地址筛选出来进行单独的处理，两者相辅相成、不可或缺。

qemu注册时需要一个EventNotifier的结构体，这个结构体保存了一个rfd和一个wfd。当系统支持eventfd的时候，两者都等于eventfd系统调用返回的fd；若系统支持，则qemu会自己利用pipe模拟eventfd，此时两者就分别是pipe的两个端口。完成了EventNotifier的注册之后，qemu利用memory_region_add_eventfd来将MemoryRegion和eventfd（此时已经包装成了EventNotifier）绑定。

MemoryRegion中有指针用于保存MemoryRegion对应的全部ioeventfd，这里所做的就是将空间进行扩充，然后在最末位加上当前要加上的eventfd。完成这一步之后，MemoryRegion就已经和对应的某一个eventfd绑定起来了。这里值得注意的是函数参数match_data，这个参数表示是否需要匹配写入的值，如果为true，则必须要虚拟机往该地址写入某值的时候，才会触发ioeventfd。

kvm注册则是通过qemu的处理，调用kvm的相关系统调用，进入到kvm的代码中进行处理的。在qemu中，kvm_mem_ioeventfd_add负责MMIO的地址，而kvm_io_ioeventfd_add负责PIO的地址。

这个IO方式与此前的不同，此前的IO是异步的，也就是说，CPU在发现IO事件后，进行一段较短时间的处理之后，返回继续执行，直到IO事件完成后向CPU发送中断，CPU接收中断信号后暂停当前的处理，再处理完成了的IO事件。整个IO过程中，CPU只需要在开始前和结束后介入即可。而同步的IO则与之相反，同步IO会阻塞CPU的运行，直到IO事件完成后返回，CPU才会继续执行下一条指令。

1.4.7 总结

vCPU线程在不同的acceleration下会有不同的表现（tcg、kvm、hvf…），在不同的IO机制下也会有不同的执行方法。

最直接的IO方式就是阻塞的方式执行IO事件，这类IO事件一般需要CPU直接参与，或CPU需要等待IO事件完成之后才能进行下一步的操作，这种事件的执行也非常简单，直接在vCPU线程上调用对应的函数即可。

KVM下，如果遇到virtio事件，会进入到kvm-virtio的处理流程。kvm负责判断地址是否是eventfd的地址，如果是就会向该fd发送信号。eventfd得到信号之后状态发生改变，qemu的事件循环会对此做出响应，根据事件循环的运转流程，对eventfd对应的回调函数进行处理。

TCG下，如果遇到virtio事件，会对事件进行分类，判断事件是PIO还是MMIO，然后再判断对应的地址是否是eventfd，如果不是，则阻塞CPU，在vCPU线程上直接运行IO；如果是eventfd，则直接对fd进行修改，然后返回CPU继续执行，对应的事件会被主循环线程捕获，在主循环线程中执行相应的IO事件，在IO实现完成之后，再返回。

1.6 Worker Thread

尽管很多I/O操作可以非阻塞的完成，有一些系统调用是没有非阻塞的替代方法可用的。而且有些耗时的计算操作就是需要简单粗暴地占用CPU，我们也很难把它们分解为回调函数。Worker thread是专用线程，在主循环线程的长时间等待会导致qemu无响应，不利于qemu的使用。因此，我们可以将其负载转移到专用的worker thread上运行。当一个工作线程需要通知qemu主线程时，可以将一个pipe或一个qemu_eventfd文件描述符添加到事件循环中。worker thread 向文件描述符写入，而当文件描述符变成可读的时候，事件循环会调用回调函数。

在qemu中，worker thread的使用是非常普遍的，一般情况下，会有数百个worker thread根据情况的不同而生成，完成任务之后又会退出。

2. 不同线程的特点

  • vCPU线程不运行事件循环
  • 主循环线程和IO线程运行事件循环
  • vCPU线程可以将事件源添加到主循环或IO线程的事件循环中
  • 回调函数在主循环线程或IO线程中运行

3. 主循环线程和IO线程之间的区别

qemu提供的原生事件循环API：AioContext

常用的函数：

  1. aio_set_fd_handler()

  2. aio_set_event_handler()

  3. aio_timer_init()
  4. aio_bh_new()

主循环使用一个glib GMainContext 和两个 AioContext 事件循环。QEMU的组件可以使用任何一种事件循环的API。主循环也将其整合成了一个事件循环函数os_host_main_loop_wait，其调用qemu_poll_nx来等待事件源。这样的做法将基于glib的代码和使用QEMU原生的AioContextAPI的代码整合在了一起。

IO线程使用一个glib GMainContext和一个AioContext。AioContext使用aio_poll()的API运行，这使得一些事件循环高级功能被启用。如果需要使用glib的事件循环，GMainContext可以使用g_main_loop_run()运行，AioContext的事件源也会被包括在内。

  • 对于qemu原生的API来说，使用aio_*()的代码可以运行在主循环或者IO循环，而使用qemu_*()的老代码只能运行在主循环中。

  • 对于glib来说，所有的代码都可以运行在主循环或者IO循环中

两种循环最关键的区别在于：

  • 主循环使用的是传统的事件循环，它调用qemu_poll_ns()

  • IO循环的AioContext调用aio_poll()，这个API拥有更高级的特性和更好的性能

4. qemu的两种事件循环API

4.1 glib GMainContext

4.1.1 GMainContext

Glib是一个由三个用C语言编写的 low-level system libraries 组成的捆绑包。

GMainContext is a generalized implementation of an event loop, useful for implementing polled file I/O or event-based widget systems (such as GTK+). It is at the core of almost every GLib application.

A GMainContext has a set of GSources which are ‘attached’ to it, each of which can be thought of as an expected event with an associated callback function which will be invoked when that event is received; or equivalently as a set of file descriptors (FDs) to check. An event could be a timeout or data being received on a socket, for example.

可以看到GMainContext中包含了许多内容，其中的timeout可以为 poll 阶段服务，当超过这个时间的等待之后，线程将直接返回，以此来避免对线程长时间的阻塞。此外，GMainContext中还包含了一个GSource的列表，GSource相当于事件源。最后，GMainContext中实际还包含诸如锁之类的东西（这里已略去），这可以保证线程的安全。

4.7 ioeventfd

一般情况下，一个完整的IO过程包括从虚拟机内部返回到KVM，再到qemu，并由qemu最终进行分发这样一个同步过程，IO完成之后的返回路径与此相反。但这样的流程会有较大的效率瓶颈，并且有时候IO请求本身只是作为一个通知事件，通知KVM或qemu完成另一个具体的IO。在这种情况下，为了优化性能，qemu采用了ioeventfd的方案来处理IO。ioeventfd基于eventfd，在初始化的时候qemu将一段IO地址空间和一个eventfd相对应起来，当虚拟机写I/O地址的时候，陷入KVM，KVM直接设置该IO地址对应的fd，qemu从其事件循环返回，执行相应的处理函数。这样就绕过了qemu的IO分发过程，提高了效率。

前面提到，Linux使用select/poll/epoll的系统调用可以监听一个文件描述符fd，当fd的状态发生变化的时候，会调用相应的事件处理函数对其进行响应。但是，有时候我们可能只需要发送一个非常简短的事件通知，没有对应的具体实体，这个时候就可以直接使用eventfd。

qemu通过EventNotifier结构体的方式注册一个ioeventfd，对应使用event_notifier_init来创建。

如果可以使用eventfd，则直接使用；否则创建一个pipe，wfd用于读，rfd用于写。

如果qemu将eventfd关联到一个事件源，并且将eventfd加入到qemu的主循环中监听，那么此时qemu的poll也会返回，用来处理这个事件。

5.2 virtio设备的结构

首先我们讨论qemu中的virtio的设备结构。

CPU拥有自己的虚拟地址空间，这个空间中包含内存和各种设备。PCI设备是属于这个空间中的，但是virtio设备却没有直接包含在这个地址空间中，取而代之的是一个被称作virtio代理设备的设备挂载在PCI总线中。也就是说，virtio-blk设备是没有MemoryRegion的，而virtio代理设备则拥有自己的MemoryRegion。Guest 触发IO事件的时候，对应的地址就是virtio代理设备的地址，对应的MemoryRegion就是virtio代理设备的MemoryRegion。在代理设备中会通过virtio总线将访问分发到具体的设备。

5.3 I/O的基本流程

接下来我们详细讲解IO的基本流程。

IO事件从Guest中分发下来之后访问对应的MemoryRegion，这个过程会判断这个MemoryRegion是否绑定着某个eventfd。具体来说，在使用tcg作为accelerator时会在IO 分发的过程中进入某段修改eventfd的逻辑；在使用kvm时，kvm负责做这个判断和修改。这个过程就是下图中signify的过程。eventfd被写入信息之后，状态发生变化，这个变化会被Host OS捕捉到，具体来说就是epoll之类的系统调用接收到信号，从系统调用中返回。

epoll返回之后即对应着MainLoop的执行逻辑，返回后就是事件循环的dispatch流程。dispatch会访问AioContext中fd变化的AioHandler，调用AioHandler中的io函数（根据事件类型的不同访问不同的函数）。这个过程就会访问到eventfd绑定好的回调函数，这个回调函数就是对应的virtio读写函数。

接下来会进入一段复杂的协程逻辑，协程中根据eventfd获取到需要的结构体，调用系统调用，提交对物理设备的IO请求，这个过程就会对真正的设备fd进行操作。
